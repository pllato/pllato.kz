# ТЗ — документы 1С: налоговый учёт, договор, назначение платежа

> Для сессии в `pllato-core-crm` (живой фронт + воркер). Цель — чтобы документы
> из CRM садились в 1С так, как ждёт бухгалтер (Асем): НДС, КПН, аналитика
> доходов/себестоимости, договор, назначение платежа.

## Корень проблемы (важно понять до правок)
CRM создаёт документы в 1С **через OData**. В толстом 1С, когда бухгалтер ставит
галочку и жмёт «Заполнить», 1С сама подтягивает налоговый учёт (НУ) из
бухгалтерского (БУ). **Через OData эта процедура НЕ запускается** — 1С пишет
только те поля, что мы прислали. Поэтому СуммаНДС, субконто, счета доходов и
себестоимости, флаг КПН — всё это воркер обязан проставлять сам. «1С подтянет
сама» здесь не сработает.

## Название галочки
- Поле документа «Реализация (акты, накладные, УПД)».
- **UI 1С:** галочка **«Учитывать КПН»** (КПН = корпоративный подоходный налог,
  казахстанский НУ).
- **OData-поле:** `УчитыватьКПН` (тип Boolean). Ставить `true`.
- Рядом стоят `УчитыватьНДС` (true) и `УчитыватьАкциз` (false) — их не трогаем
  сверх текущего.

## Что уже работает в проде (НЕ ломать)
- Ставка НДС строки (`СтавкаНДС_Key`) и расчёт `СуммаНДС` формулой «в т.ч.».
- Код назначения платежа `710` на счёте (`КодНазначенияПлатежа`).
- Адрес поставки (`АдресДоставки`) из контрагента.
- Флаг `УчитыватьКПН=Да` на реализации.
- Контрагент по БИН.

## Правка 2 — СуммаНДС на счёте И на реализации (✅ выполнено)
Через OData обработчик строки не вызывается → считать самим (уже есть
`lineVatSum`, формула «в т.ч.»: `Сумма*ставка/(100+ставка)`). Расчёт применяется
и к счёту, и к реализации. Проверка: НДС 12% на 111100 → ≈ 11903.57;
НДС 5% на 2250 → 107.14; НДС 16% на 2250 → 310.34.

## Правка 2b — БАГ: СЧЁТ теряет выбранную ставку НДС (❗ открыто)
**Факт (22.06.2026):** в заказе выбрана ставка **16%**. Результат:
- Реализация №532 строка — **16%** (`СтавкаНДС_Key=fecafe34…`), СумНДС 310.34 — **верно**.
- Счёт №381 строка — **5%** (`СтавкаНДС_Key=34dffed7…` = дефолт `ONE_C_VAT_5`),
  СумНДС 107.14 — **неверно**. Карточка тут ни при чём (её дефолт 12%).

**Причина:** счёт и реализация берут ставку из разных мест. Реализация — из
строки заказа (правильно). Счёт — из `deal.oneCVatRef || ONE_C_VAT_5`, а
выбранная ставка до шапки сделки **не доходит/не сохраняется** (персист
`oneCVatRef` только по событию `change`, см. правку фронта #1) → счёт падает на
дефолт 5%.

**Что сделать:**
1. Счёт должен брать **ту же построчную ставку НДС, что и реализация**
   (из строки заказа), а не из `deal.oneCVatRef`.
2. Починить персист выбранной ставки на сделке (правка фронта #1): сохранять
   `oneCVatRef` не только на `change`, но и при сабмите/blur.
3. Сменить дефолт-фолбэк `ONE_C_VAT_5` → **12%** (стандарт КЗ), чтобы при сбое
   не получались «левые» 5%.

**Проверка:** заказ со ставкой 16% → и счёт, и реализация строкой = 16%,
СумНДС = 310.34 на сумму 2250.

## Правка 3 — договор: выбор в форме + основной договор
- Бэкенд уже принимает `contractRef`; если пуст — берёт «Основной договор
  контрагента» (`ОсновнойДоговорКонтрагента_Key`), если и его нет → пусто.
- Добавить в форму заказа **селектор договора** контрагента (передавать
  `contractRef`).
- Параллельно — проследить, чтобы у контрагентов в 1С был заполнен основной
  договор (иначе fallback пуст, как у «ТОО Платон»).

Если бухгалтеру нужно выбирать — добавить в форму выпадающий **710 / 859**

## Правка 5 — себестоимость (отдельная задача, не правка полей)
Себестоимость — это **не число, а счёт списания**; сумму 1С берёт **при
проведении из партий/закупок**. Даже с верным счётом списания сумма = 0, пока по
новым карточкам нет прихода/остатка с себестоимостью. → Завести в 1С
приход/остатки с закупочной ценой по новым карточкам (нужны закупочные цены от
Асем). Это закрывает «себестоимость не тянет» по сути, а не по полю.

## Как принимать работу (с Асем, по одному документу)
1. Создать заказ из нового каталога → провести Счёт → Реализацию → Счёт-фактуру.
2. Открыть проведённую реализацию, нажать **Дт/Кт (проводки)** и вместе с Асем
   сверить: счёт доходов, номгруппа, статья дохода, КПН, счёт и аналитика
   себестоимости. Сверять по **проводкам**, а не только по табличным полям.
3. Счёт — проверить «Назначение платежа». Счёт-фактура — основание = реализация,
   суммы/НДС совпадают.

## Гочи
- Через OData 1С НЕ дозаполняет НУ из БУ — слать всё явно.
- GUID-ы плана счетов/субконто РАЗНЫЕ у каждой базы — блок только для `aminamed`.
- 1С:Фреш отдаёт 402 при частых/тяжёлых запросах — не спамить.
