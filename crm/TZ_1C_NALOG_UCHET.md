# ТЗ — документы 1С: налоговый учёт, договор, назначение платежа

> Для сессии в `pllato-core-crm` (живой фронт + воркер). Цель — чтобы документы
> из CRM садились в 1С так, как ждёт бухгалтер (Асем): НДС, КПН, аналитика
> доходов/себестоимости, договор, назначение платежа.

## Корень проблемы (важно понять до правок)
CRM создаёт документы в 1С **через OData**. В толстом 1С, когда бухгалтер ставит
галочку и жмёт «Заполнить», 1С сама подтягивает налоговый учёт (НУ) из
бухгалтерского (БУ). **Через OData эта процедура НЕ запускается** — 1С пишет
только те поля, что мы прислали. Поэтому СуммаНДС, субконто, счета доходов и
себестоимости, флаг КПН — всё это воркер обязан проставлять сам. «1С подтянет
сама» здесь не сработает.

## Название галочки
- Поле документа «Реализация (акты, накладные, УПД)».
- **UI 1С:** галочка **«Учитывать КПН»** (КПН = корпоративный подоходный налог,
  казахстанский НУ).
- **OData-поле:** `УчитыватьКПН` (тип Boolean). Ставить `true`.
- Рядом стоят `УчитыватьНДС` (true) и `УчитыватьАкциз` (false) — их не трогаем
  сверх текущего.

## Правка 2 — СуммаНДС на счёте И на реализации (✅ выполнено)
Через OData обработчик строки не вызывается → считать самим (уже есть
`lineVatSum`, формула «в т.ч.»: `Сумма*ставка/(100+ставка)`). Расчёт применяется
и к счёту, и к реализации. Проверка: НДС 12% на 111100 → ≈ 11903.57;
НДС 5% на 2250 → 107.14; НДС 16% на 2250 → 310.34.

## Правка 2b — СЧЁТ терял выбранную ставку НДС (✅ ВЫПОЛНЕНО 22.06.2026)
**Было:** в заказе выбрана 16%, реализация брала 16% (верно), а счёт падал на
дефолт 5% (`ONE_C_VAT_5`) — рассинхрон.
**Стало (проверка 15:28):** заказ со ставкой 16% → и счёт, и реализация строкой
= **16%**, СумНДС **206.9** на сумму 1500. Рассинхрон устранён.
Ниже — исходный диагноз (оставлен для истории):

**Причина:** счёт и реализация берут ставку из разных мест. Реализация — из
строки заказа (правильно). Счёт — из `deal.oneCVatRef || ONE_C_VAT_5`, а
выбранная ставка до шапки сделки **не доходит/не сохраняется** (персист
`oneCVatRef` только по событию `change`, см. правку фронта #1) → счёт падает на
дефолт 5%.

**Что сделать:**
1. Счёт должен брать **ту же построчную ставку НДС, что и реализация**
   (из строки заказа), а не из `deal.oneCVatRef`.
2. Починить персист выбранной ставки на сделке (правка фронта #1): сохранять
   `oneCVatRef` не только на `change`, но и при сабмите/blur.
3. Сменить дефолт-фолбэк `ONE_C_VAT_5` → **12%** (стандарт КЗ), чтобы при сбое
   не получались «левые» 5%.

**Проверка:** заказ со ставкой 16% → и счёт, и реализация строкой = 16%,
СумНДС = 310.34 на сумму 2250.

## Правка 3 — договор: выбор в форме + данные в 1С
**Это НЕ баг кода.** Бэкенд уже подставляет «Основной договор контрагента»
(`ОсновнойДоговорКонтрагента_Key`) автоматически, когда он у контрагента задан.
Проверено: на реализациях №529/№530 (контрагент «Городская поликлиника №33»)
договор подставился; пусто только там, где у контрагента договоров нет.

**Масштаб по базе Аминамед (перепроверено 23.06.2026 по `Ref_Key`, надёжно):**
- Договор есть **почти у всех** контрагентов (из 30 проверенных продажных — у
  всех 30 задан основной договор). Контрагенты без договора — **исключение**
  (напр. «ТОО Платон»).
- Договор вида **«С покупателем»** (корректный для продаж) — у **89** контрагентов
  (всего таких договоров 266). Остальное — «Прочее» (3299) / «С поставщиком» (21).
- ⚠️ Вчерашние цифры «79% / 456 дублей / ТОО Платон ×4» были ЗАВЫШЕНЫ артефактом
  `$skip`-пагинации 1С — им не верить. Реальные дубли карточек номенклатуры
  **отсутствуют** (384/322 — ровно как заводили). Дубли контрагентов существуют,
  но точечно (Платон — 3 карточки).
- Кандидаты с основным договором «С покупателем» (для теста): **АПТЕКА ПЛЮС,
  ТИН, Алматинский онкологический центр, ТОО «HELMIR», Медицинский центр Евразия**.

**Что сделать:**
1. **Форма заказа:** добавить **селектор договора** контрагента (передавать
   `contractRef`) — чтобы выбирать конкретный договор, когда их несколько, и
   видеть, что договора нет.
2. **Данные в 1С (отдельно от кода):** у контрагентов без договора (21%) завести
   договор / проставить основной; почистить **дубли карточек** (особенно тех, что
   используются в заказах, — «ТОО Платон» и т.п.), иначе матч по БИН может сесть
   на пустой дубль.
3. Бэкенд-фолбэк «основной договор» уже работает — не трогать.

## Правка 3b — АВТОСОЗДАНИЕ договора, если его нет (готовая схема)
Логика для воркера (база Аминамед): при создании документа резолвим договор;
если у контрагента договора нет (`ОсновнойДоговорКонтрагента_Key` пуст И активных
договоров 0) — **создаём договор «С покупателем»** через OData, берём его
`Ref_Key` как `ДоговорКонтрагента_Key` документа. Опц.: PATCH-нуть контрагента,
проставив новый договор основным.

**Коллекция:** `Catalog_ДоговорыКонтрагентов`. **POST ниже проверен — HTTP 201**
(создан и удалён 23.06.2026). `Code` НЕ слать — 1С присвоит сама.

```json
POST /odata/standard.odata/Catalog_ДоговорыКонтрагентов
{
  "Owner_Key": "<Ref_Key контрагента>",          // ОБЯЗАТЕЛЬНО
  "Description": "Договор с покупателем",          // имя договора (свободно)
  "ВидДоговора": "СПокупателем",                   // enum: СПокупателем|СПоставщиком|Прочее
  "Организация_Key": "8678efaa-9684-4325-a198-7f3c8a1bc2f3",   // Аминамед (per base!)
  "ВалютаВзаиморасчетов_Key": "9e9a6ffb-aa56-11e1-b9c4-002215ba1bbe", // KZT (per base!)
  "ВедениеВзаиморасчетов": "ПоДоговоруВЦелом",
  "ДатаДоговора": "2026-06-23T00:00:00",
  "ДатаНачалаДействияДоговора": "2026-06-23T00:00:00",
  "НомерДоговора": "CRM-AUTO-1"
}
// → 201, ответ содержит Ref_Key — его и подставлять в документ как
//   payload.ДоговорКонтрагента_Key
```

**Опц. сделать основным** (чтобы и в 1С подхватывался):
```
PATCH Catalog_Контрагенты(guid'<owner>')
{ "ОсновнойДоговорКонтрагента_Key": "<Ref_Key нового договора>" }
```

**Значения per base** (для ИП Баймуханова `/a/ea68/264980` — СВОИ):
- Организация_Key: `f9f0a501-a9ed-11ee-9866-f8b15698efb3`
- ВалютаВзаиморасчетов_Key: `dda1de7d-a9ed-11ee-9866-f8b15698efb3`

**Идемпотентность:** перед созданием проверять, нет ли уже договора (чтобы не
плодить дубли при повторной выгрузке) — искать активный договор по `Owner_Key`,
и только если нет — создавать.

## Правка 5 — себестоимость (отдельная задача, не правка полей)
Себестоимость — это **не число, а счёт списания**; сумму 1С берёт **при
проведении из партий/закупок**. Даже с верным счётом списания сумма = 0, пока по
новым карточкам нет прихода/остатка с себестоимостью. → Завести в 1С
приход/остатки с закупочной ценой по новым карточкам (нужны закупочные цены от
Асем). Это закрывает «себестоимость не тянет» по сути, а не по полю.

## Как принимать работу (с Асем, по одному документу)
1. Создать заказ из нового каталога → провести Счёт → Реализацию → Счёт-фактуру.
2. Открыть проведённую реализацию, нажать **Дт/Кт (проводки)** и вместе с Асем
   сверить: счёт доходов, номгруппа, статья дохода, КПН, счёт и аналитика
   себестоимости. Сверять по **проводкам**, а не только по табличным полям.
3. Счёт — проверить «Назначение платежа». Счёт-фактура — основание = реализация,
   суммы/НДС совпадают.

## Гочи
- Через OData 1С НЕ дозаполняет НУ из БУ — слать всё явно.
- GUID-ы плана счетов/субконто РАЗНЫЕ у каждой базы — блок только для `aminamed`.
- 1С:Фреш отдаёт 402 при частых/тяжёлых запросах — не спамить.
